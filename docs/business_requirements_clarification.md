# 业务需求澄清记录

本文用于承接“澄清业务需求”会话。当前无法直接读取历史聊天原文，因此以下内容基于仓库中的 `business_requirements.md`、`lesson_1.md`、`system_architecture_initial_design.md`、现有脚本和 beads 备份恢复。

## 1. 已确认的业务目标

1. 建设一个高频电商价格指数计算平台，用电商 SKU 日价格数据补充传统 CPI 的低频、粗粒度问题。
2. 支持 T+1 每日出表，即今天产出昨天的价格指数。
3. 支持至少两个展示层级：
   - 总指数：全网整体价格走势。
   - 类目指数：一级类目价格走势。
4. 最终交付物需要能生成趋势折线图，当前为：
   - `output/total_index_trend.png`
   - `output/category_index_trend.png`
   - `output/price_index_daily.csv`
5. 课程工程约束要求架构轻量、低运维、低耦合，因此当前采用“对象存储 + ClickHouse + Python”的方案，而不是 Hadoop、Spark、Kafka 等重型集群方案。

## 2. 当前实现口径

当前代码已经从设计阶段推进到可运行数据管道：

1. 原始数据通过 OSS 的 `raw/` 路径存储。
2. `scripts/load_from_oss_to_clickhouse.py` 从 OSS 读取商品、类目和日价格 CSV，并写入 ClickHouse。
3. 日价格清洗口径：
   - 丢弃缺少 `product_id` 或 `price` 的记录。
   - 丢弃 `price <= 0` 的记录。
   - 同一日同一 SKU 多条报价使用杰文斯几何平均去重。
4. `scripts/compute_index.py` 在 ClickHouse 内计算指数。
5. `scripts/plot_index.py` 从结果表生成总指数和类目指数折线图。
6. GitHub Actions 已支持手动触发从 OSS 导入 ClickHouse、计算指数、生成图表并上传结果。

## 3. 指数算法澄清

设计文档中曾提出保存拉氏指数、派氏指数和费雪指数。

但当前样例数据没有逐日销量字段，只有商品和类目维表中的固定 `weight`。在这个数据条件下：

1. 当前主指标应明确为拉氏指数。
2. 权重采用维表固定权重，而不是每日销量权重。
3. 费雪指数依赖当期销量权重；如果没有逐日销量，费雪指数会退化或无法严格成立。
4. 因此当前实现不单独输出费雪指数是合理的，但需要在报告中说明原因，避免“设计承诺”和“实现交付”不一致。
