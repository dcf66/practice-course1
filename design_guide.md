# 高频电商价格指数：系统设计与开发考量指南

在正式动手写代码之前，一份严谨的架构设计是项目成功的保障。本文档整理了本项目在落地过程中的五大核心设计考量点，供大家在项目实操、技术选型及代码编写时参考。

---

## 1. 怎么进行模块化设计？

**核心理念：高内聚，低耦合。** 
在一个完整的大数据系统中，绝对不能把所有逻辑揉在一个几千行的 Python 脚本中。本项目建议遵循经典的“数据流水线”思想，将系统划分为以下四个核心模块：

- **数据接入与清洗层（Data Ingestion & ETL）**：
  - **职责**：负责读取外部的原始电商日志数据。处理异常值（如价格为负数或0）、处理缺失值（如某个 SKU 今天没有报价），并进行格式标准化。
- **数据存储层（Data Storage）**：
  - **职责**：屏蔽底层存储介质的差异。封装向对象存储（MinIO/OSS）上传/下载原始数据的方法，以及向分析型数据库（ClickHouse）读写数据的方法。
- **核心算法引擎层（Compute Engine）**：
  - **职责**：这是一个纯计算模块，不应该包含任何数据库连接代码。它接收标准的输入（如 Pandas DataFrame），根据指定的数学公式（拉氏、费雪等）计算出指数，并返回结果。
- **自动化与调度层（Orchestration）**：
  - **职责**：作为流程的“大脑”，按照时间线依次调用“拉取 -> 清洗 -> 计算 -> 存储”模块。在企业级项目中，通常由 Airflow 等工具接管，在本课程中可由一个主控 Python 脚本通过定时任务触发。

## 2. 怎么组织工程目录结构（物理划分）？

有了逻辑上的模块划分，在物理层面（文件系统）我们也需要有严谨的目录规范。推荐采用以下标准的数据工程目录结构：

- **`src/` (核心源代码目录)**：
  存放所有的业务逻辑代码。按照模块化思想，内部可进一步划分为 `src/ingestion` (数据采集与清洗)、`src/storage` (数据库交互逻辑)、`src/compute` (指数算法引擎) 等子文件夹。
- **`config/` (配置文件目录)**：
  存放项目中所有**非敏感**的变动参数，如 `.yaml` 或 `.json` 文件。例如数据库的 IP 端口、ClickHouse 的表名、任务调度的定时参数等。**绝对不要把配置硬编码（Hardcode）在代码里**，否则切换开发/生产环境时将是一场灾难。
- **`.env` (环境变量文件)**：
  专门存放**敏感配置**（如数据库密码、云平台 Secret Key）。**该文件绝对不能提交到 Git 代码库中**（必须配置在 `.gitignore` 里）。代码运行时，通过 `python-dotenv` 等库将密码加载到环境变量中，从而实现安全解耦。
- **`tests/` (测试代码目录)**：
  独立于 `src/` 之外。存放所有的单元测试和集成测试代码（例如使用 `pytest` 编写的 `test_compute.py`）。这样可以保证在打包上线时，不会把测试用的冗余代码混入核心业务中。

## 3. 怎么选型具体技术方案？

选型的本质是**权衡（Trade-off）**。我们需要根据数据的“体量”和“访问特征”来做决定：

- **为何引入对象存储（如 MinIO/S3）？**
  - **考量**：每日千万级的数据，如果全量放在传统数据库中，成本极其昂贵。对象存储提供了廉价且容量无限的“数据湖”底座，非常适合作为海量原始数据（Raw Data）的备份归档仓库。
- **为何使用列式数据库（如 ClickHouse）？**
  - **考量**：在计算指数时，我们通常只需要读取部分列（如 Date, Price, Sales），并且需要对这些列做极其频繁的全局或分组聚合（Group By 类目）。传统的行式数据库（如 MySQL）在扫表聚合时性能会极速衰退，而 ClickHouse 这种专门为 OLAP（在线分析处理）设计的列式数据库能提供惊人的秒级响应速度。
- **为何使用 Python 作为主语言？**
  - **考量**：在数据科学与大数据调度领域，Python 拥有最完善的生态。无论是数据清洗（Pandas）、对象存储交互（Boto3），还是 ClickHouse 连接（clickhouse-driver），都可以极速实现业务逻辑。

## 4. 怎么设计数据表（Schema）？

在列式数据库中，Schema 设计的好坏直接决定了查询速度。

**① 字段设计（包含什么）**
- **明细表（Raw Data）**：`date` (日期), `sku_id` (商品唯一标识), `category` (细分类目), `price` (当日均价), `sales` (当日销量或销售额)。
- **指数结果表（Index Result）**：`date` (计算日期), `index_level` (层级，如“总指数”或“类目指数”), `category_name` (具体的类目名称), `index_type` (算法类型，如拉氏/费雪), `index_value` (最终计算得出的指数值)。

**② 性能设计（如何快）**
以 ClickHouse 的建表逻辑为例，必须慎重考虑以下几点：
- **排序键（Order By Key）**：数据在硬盘上是按排序键物理排序的。由于我们几乎所有的计算都是基于“按天”和“按类目”筛选的，所以 `ORDER BY (date, category, sku_id)` 是比较理想的设计，能让引擎迅速跳过无关数据。
- **分区键（Partition By）**：海量数据需要切割成一个个小块管理。电商数据最典型的分区方式是**按月（toYYYYMM(date)）**或者**按天**分区，避免单个查询扫描过多的历史文件。

## 5. 怎么选择核心计算算法？

我们在项目中不仅要算出一个数，更要保证这个数具备经济学上的合理性。

- **底层 SKU 去重与清洗算法**：
  同一款 SKU 一天内可能有多个报价（不同商家或不同时间点）。在底层汇总时，可以采用**杰文斯指数（Jevons Index，无权重的几何平均）**，这能有效抹平某些商家的极端异常高价或低价。
- **聚合层（从 SKU 到最终指数）**：
  - **初级方案：拉斯贝尔指数（拉氏指数）**。这是最传统的做法，采用固定（基期）的销量作为权重进行计算。实现简单，但缺点是在市场发生剧变时会高估物价变动（因为忽略了消费者的替代效应）。
  - **高阶推荐：费雪理想指数（Fisher Index）**。既然电商数据能够提供实时的每日销量，我们就应当充分利用。结合基期销量（拉氏）与当期销量（派氏）求几何平均得出费雪指数。这能最真实地反映高频动态市场下的物价波动。

## 6. 怎么设计测试代码？

在大数据计算中，如果逻辑错了，结果往往只是一个偏差的数字，系统并不会报错。因此，测试设计尤为关键。

- **单元测试（Unit Testing，针对计算逻辑）**：
  - **做法**：不要使用真实的千万级数据集测试算法！你需要手动“捏造”极小规模的边界数据集（例如：只有3个商品，其中1个销量突然跌至0，另1个价格暴涨10倍）。
  - **目的**：通过 `pytest` 等框架断言你的指数计算函数（如费雪指数公式）在这类极端场景下是否会报除零错，输出的数值是否符合纸笔演算的数学预期。
- **集成测试（Integration Testing，针对数据流转）**：
  - **做法**：利用 Docker 快速启动一套轻量级的 MinIO + ClickHouse 环境。写入一百条模拟日志，运行自动化脚本，检查数据是否能成功流经“清洗 -> 上传 -> 落库 -> 聚合”四个阶段，最终落在结果表中。
- **数据质量监控（Data Quality Checks，运行时防御）**：
  - **做法**：在调度代码中埋入监控点。例如，计算出今天的总指数后，必须与昨天的指数进行比对。如果单日涨跌幅超过 50%（电商总物价一天翻倍是不可能的），系统应当立即告警并**熔断**后续的发布流程，这说明源数据极有可能被污染了。
